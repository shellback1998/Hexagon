Implementing a Delivered Piping Spec
  • Two types of information required to initiate a piping specification: Spec data and spec rules
  • Spec data include:  Spec name, materials of construction, fluid service, design standard, etc
  • Spec rules include:  Service limits, NPD size table, branch table, weld clearance, etc.


  • See example of loading a spec’s piping materials class data as delivered by Intergraph with spec rules

Implementing Delivered Piping Part Data I
  • Delivered part data can be loaded to catalog as-is
  • Delivered part data can be modified then loaded to catalog
  • Part data include: Short code, option code, size, commodity code, schedule, rating, face to face, face to center, etc.
  • Part spec rules include: Port alignment, min pipe length, field fit length, etc.
  • Notice that part rules worksheets do not have a “SpecName” column like spec rules do

Creating a Custom Piping Spec

    • In “CatalogData” workbook, locate piping parts needed, copy and modify as needed, most often commodity code and dimensions are adjusted per the new spec requirements
    • Bulkload workbooks to catalog
    • Run spec Verification and review report
    • “Allow” spec for modeling and perform preliminary manual testing
    • Repair, rebulkload and retest until all parts in the spec place as needed

Working With Code Listed Values
  • A code listed value provides a standard attribute linked to a numeral that uniquely identifies that attribute
  • Spec writer can add new code lists
  • When modifying a code list, the spec writer must follow requirements for that list, e.g. some code lists require user-added values to be greater than 10,000; other code lists allow changing textual value but not numeric value or meaning

Defining Instrument Items
  • Two types of Instruments: Catalog & Custom
  • Catalog instruments are fully defined like regular spec components
  • Catalog instruments can be accessed by entering their tag or by selecting them from a P&ID drawing
  • Custom “On-the-fly” instruments are partially defined in the catalog, the user selects them from a menu and then finishes defining dimensions and attributes
  • Custom instruments can be read in from INtools in an integrated environment

Basic Spec Troubleshooting
  • Three main areas of troubleshooting:
    • Rules
    • Spec data
    • Part data
  • Two ways of diagnosing a spec:
    • Using Validation report: Always run after loading a spec. May have some false hits that can be ignored
    • Using manual placement: Verifies problems found by Validation process, plus may find additional issues passed by Validation such as incorrect parameters
19
Spec Rules Troubleshooting
  • Some rules can be turned off by simply not defining them in the spec. For example, to avoid the Temp/Pressure limits check resulting in the error below, the user can:
    • Delete the spec’s pressure/temp. limits table from the catalog, or
    • Set UndefinedServiceLimitsruleOpt in DefaultProjectOptions


  • Some rules generate To-Do entries, like pipe length violations. During modeling testing always review the To-Do list <CTL-T> for rule related errors or warnings

Spec Rules Troubleshooting
  • Two groups of rules:
    • Piping Specification Rules
    • Piping Materials Class Rules. First column is always “SpecName”
  • Spec rules data defined in:
    • File: <NamePrefix>_SpecificationData.xls
21
Spec Data Troubleshooting
  • Spec attribute data is defined in:
    • File: <NamePrefix>_SpecificationData.xls
    • Worksheet: PipingMaterialsClassData
  • Only the SpecName attribute is required
  • Attributes under the blue columns are optional, but at least description and service should be defined
  • Any problems with specification attributes information can be corrected by editing this worksheet
22
Part Data Troubleshooting
  • This error indicates that a piping part was defined in the PipingCommodityFilter and PipingCommodityMatlControlData worksheets, but there is no catalog information about it containing end prep and symbol dimensional data
  • File: <NamePrefix>_CatalogData.xls
  • Worksheet: Part Class specific, i.e. Union, 45DegElbow, GlobeValve, OrificeFlange, etc.
  • Action: Match commodity code and NPD from commodity filter and matl ctl data worksheets with same attributes in Part Class worksheet
  • Beware: For valves, the valve body data may be correct, but problems with the operator data, i.e. bad commodity code, may still cause this error
23
Part Data Troubleshooting
  • This error indicates that a piping part was defined in the PipingCommodityFilter but no matching data is found in PipingCommodityMatlControlData worksheet
  • File: <NamePrefix>_SpecificationData.xls
  • Worksheet: PipingCommodityMatlControlData
  • Action: Match component’s commodity code from piping commodity filter with matl ctl data worksheet, then validate all standard note attributes for component in this worksheet
24
Part Data Troubleshooting
  • This error indicates that piping part dimensional values passed from the spec to the symbol graphics placement executable did not satisfy the required inputs needed to draw the graphic
  • File: <NamePrefix>_CatalogData.xls
  • Worksheets:
    • Part Class specific, i.e. Union, 45DegElbow, etc.
    • Termination type specific, i.e. PipingGenericDataFemale
  • Action:
    • Check body dimensions: FacetoCenter, FacetoFace, etc.
    • Match part’s applicable end prep atts (Press. Rating, End Prep, End Std, Schedule) with input attributes in “GenericData” worksheets to make sure that end prep dimensions are found
25
Implied Data Troubleshooting
  • This error indicates that part placement will be successful, but the implied gasket(s) for the bolted end(s) are not found in the catalog
  • File: <NamePrefix>_SpecificationData.xls
  • Worksheet: GasketSelectionFilter, DefaultProjectOptions
  • File: <NamePrefix>_CatalogData.xls
  • Worksheet: GasketPartData
  • Action:
    • Verify that part’s end prep data (Press. Rating, End Prep, End Std) have a match in the inputs (yellow) section of the GasketSelectionFilter worksheet
    • Make sure the gasket commodity code in GasketSelectionFilter has a match in GasketPartData worksheet
    • Set GsktSlnForMismatchBltdEndsOpt option = 10 in Default-ProjectOptions to allow for mismatched end preps
26
Implied Data Troubleshooting
  • Bolted parts will place successfully before bolt data is fully defined. Reviewing the Properties>connection relationships will show a line for Bolts if they are properly defined. If bolt data is missing do as follows
  • File: <NamePrefix>_SpecificationData.xls
  • Worksheet: BoltSelectionFilter, DefaultProjectOptions
  • File: <NamePrefix>_CatalogData.xls
  • Worksheet: BoltPartData
  • Action:
    • In DefaultProjectOptions make sure that BoltLengthCalculationOption = 5 (Enabled)
    • Verify that part’s end prep data (Press. Rating, End Prep, End Std)  have a match in the inputs (yellow) section of the BoltSelectionFilter worksheet
    • Make sure the bolt commodity code in BoltSelectionFilter has a match in BoltPartData worksheet
27
Instrument Part Troubleshooting
  • This warning indicates that placement cannot proceed because the entered instrument does not exist in the catalog, or does not exist for the active NPD and/or end preparation at the active point. To resolve this do as follows
  • File: InstrumentData.xls
  • Worksheet: Instrument part class (ie FlowMeter1), InstrumentClassData, PipingCommodityMatlControlData
  • Action:
    • Make sure that the attribute “RequisitionType” is defined in both the instrument part and InstrumentClassData worksheets
    • Make sure that the IndustryCommodityCode from the part sheet is matched in the ContractorCommodityCode columns of InstrumentClassData and PipingCommodityMatlControlData
    • If RequisitionType = 5, ensure that the instrument data is defined in PipingCommodityMatlControlData just like a regular piping component
